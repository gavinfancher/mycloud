# 00 — Architecture & Decisions

Read this before any phase. It defines the system shape, naming, the two network planes, the
state model, and the key technical decisions.

## 1. Components

```
                          ┌──────────────────────── Cloudflare (myhomecloud.dev) ───────────────────────┐
   Off-tailnet user ──────► DNS (CNAME, proxied) ──► Cloudflare Tunnel ──► (encrypted) ──┐               │
                          └────────────────────────────────────────────────────────────┼───────────────┘
                                                                                          │
                                                            ┌─────────────── Control node (a Proxmox VM) ───────────────┐
                                                            │                                                            │
   On-tailnet user ──► Tailscale split DNS for ────────────► CoreDNS (resolver)   Caddy (reverse proxy) ◄── cloudflared │
                       myhomecloud.dev → control node        │  answers A = instance tailnet IP    │                    │
                                                            │                                       │                    │
                                                            │   homecloud controller (FastAPI + UI + jobs)              │
                                                            │     - Proxmox client   - Cloudflare DNS                    │
                                                            │     - Tailscale client - Caddy config writer              │
                                                            │     - port scanner     - CoreDNS zone writer              │
                                                            └───────────────────────────┬────────────────────────────────┘
                                                                                         │ Proxmox API + SSH
                                                                                         ▼
                                                       ┌──────────────── Proxmox host (pve-root) ────────────────┐
                                                       │   instance "app"   instance "db"   ...   (join tailnet)  │
                                                       └──────────────────────────────────────────────────────────┘
```

The control node runs three cooperating processes (all as containers via one
`docker-compose.yml` on the control node):

- **controller** — the FastAPI app in this repo (UI + API + jobs).
- **caddy** — HTTP reverse proxy; config files generated by the controller.
- **cloudflared** — Cloudflare Tunnel connector; routes public hostnames to Caddy.
- **coredns** — authoritative resolver for `myhomecloud.dev` on the tailnet; zone file
  generated by the controller. (Phase 06.)

## 2. The two network planes

### Public web plane (off-tailnet)

- Cloudflare holds the zone for `myhomecloud.dev`.
- For each **published web service**, the controller creates a **CNAME → `<tunnel-id>.cfargotunnel.com`**, proxied (orange cloud). TLS is terminated at the Cloudflare edge.
- The Cloudflare Tunnel ingress sends **all** hostnames to Caddy on the control node (`http://caddy:80`).
- Caddy routes by `Host` header to the instance's **tailnet IP : port**. (The control node is on the tailnet, so it can reach every instance privately.)
- Hostnames:
  - `app.myhomecloud.dev` → the instance's primary web port (default 80, configurable).
  - `<service>.app.myhomecloud.dev` → a specific detected port on the instance.

### Private plane (on-tailnet — databases & arbitrary TCP)

- Every instance joins the tailnet (already implemented). On the tailnet, **all** of an
  instance's listening TCP ports are reachable at `<instance>.<tailnet>.ts.net:<port>` with no
  extra config.
- To use friendly `myhomecloud.dev` names privately, run **CoreDNS** on the control node as an
  authoritative server for `myhomecloud.dev`, returning **A records = instance tailnet IPs**.
- Configure **Tailscale split DNS** (admin console → DNS → "Nameservers" → "Add nameserver" →
  "Restrict to domain" `myhomecloud.dev`, IP = control node tailnet IP). Now tailnet devices
  resolve `db.myhomecloud.dev` (or `db.app.myhomecloud.dev`) to the instance's tailnet IP and
  connect **directly over WireGuard** — including raw TCP like Postgres.
- **Databases never get a Cloudflare record**, so they are unreachable off-tailnet. Privacy is
  the default; publishing to the web is an explicit action.

### Why the same name can be both

`app.myhomecloud.dev` may exist as a Cloudflare CNAME (public web via tunnel) **and** be
answered by CoreDNS with the tailnet IP (private, all ports). Tailscale split DNS takes
precedence for tailnet devices, so on the tailnet you get the direct private IP; elsewhere you
get the public tunnel (web only). This is exactly the desired behavior.

> Constraint: Cloudflare does not proxy multi-label wildcards (`*.app.myhomecloud.dev`) on
> non-Enterprise plans. We therefore create an explicit record per published service. CoreDNS
> (private side) can use wildcards freely.

> **Cloudflare Access:** the zone sits behind Cloudflare Access. Published hostnames (via the
> tunnel) are gated by Access policies at the edge — publishing a service makes it reachable
> *to authenticated identities*, not the open internet. Cover new hostnames with a
> self-hosted Access app on `*.myhomecloud.dev` so each published service is protected without
> per-service setup. Machine clients use an Access **service token** (or a scoped bypass).
> The controller's own console stays on the tailnet and is not behind Access.

## 3. Naming model

- Instance name: `^[a-z][a-z0-9-]{1,30}$` (already enforced).
- Tailnet hostname == instance name (already implemented) → `app.<tailnet>.ts.net`.
- Public/base web host: `app.myhomecloud.dev`.
- Per-service web host: `<service>.app.myhomecloud.dev` where `<service>` is a user-chosen
  label for a detected port (e.g. `grafana.app.myhomecloud.dev` → `app:3000`).
- Private DB host: use the same `app.myhomecloud.dev` (resolves to tailnet IP on-tailnet) or an
  optional alias `db.app.myhomecloud.dev`; connect on the DB port over the tailnet.

## 5. Configuration additions (`src/homecloud/config.py`)

Add these settings (all optional, with safe empty defaults so dev runs keep working):

```python
# Public domain
domain: str = "myhomecloud.dev"

# Cloudflare
cloudflare_api_token: str = ""
cloudflare_zone_id: str = ""
cloudflare_tunnel_id: str = ""           # <id>.cfargotunnel.com is the CNAME target
cloudflare_tunnel_cname: str = ""        # convenience: full "<id>.cfargotunnel.com"

# Caddy
caddy_config_dir: str = "/etc/caddy/sites"   # controller writes *.caddy here (mounted volume)
caddy_reload_cmd: str = ""                    # e.g. "docker exec caddy caddy reload --config /etc/caddy/Caddyfile"

# Local resolver (split DNS)
coredns_zone_path: str = "/etc/coredns/db.myhomecloud.dev"
coredns_reload_cmd: str = ""                   # CoreDNS auto-reloads the zone file; usually empty
control_node_tailscale_ip: str = ""            # used in SOA/NS and as default upstream host

# Default primary web port for an instance's base hostname
default_web_port: int = 80
```

Update `.env.example` accordingly. Keep the existing Proxmox/Tailscale settings.

## 6. Decisions (defaults chosen)

1. **Public exposure = Cloudflare Tunnel.** No router/firewall changes; works behind NAT;
   per-hostname routing; edge TLS. Alternative (port-forward + wildcard cert) is documented but
   not implemented.
2. **Reverse proxy = Caddy**, config-file-per-service in a mounted dir; reload via
   `caddy reload`. Reuse the logic in the stale `src/homecloud/proxy/caddy.py` (Phase 04 fixes
   it to match the new settings).
3. **Resolver = CoreDNS** with the `file` plugin and `auto`/`reload` so a regenerated zone file
   is picked up without a restart. Alternatives (dnsmasq, Technitium) are acceptable but the
   plan is written for CoreDNS.
4. **Upstream from Caddy = instance tailnet IP**, since the control node is on the tailnet and
   this needs no extra Proxmox networking.
5. **Per-service records** (not wildcards) on Cloudflare due to the multi-label wildcard
   limitation.
6. **All infra runs on the control node** via one `docker-compose.yml`, with volumes shared
   between the controller (writer) and caddy/coredns (readers).
