# 06 — Private split DNS (databases & TCP over Tailscale)

## Objective

Make `myhomecloud.dev` names resolve to **instance tailnet IPs** when you are on the tailnet,
so databases and arbitrary TCP services are reachable privately by name without any public
exposure.

## Context

- Every instance already joins the tailnet; all its TCP ports are reachable at
  `<instance>.<tailnet>.ts.net:<port>`. The goal here is friendly `*.myhomecloud.dev` names on
  the private side and a single source of truth for naming.
- `src/homecloud/tailscale/ssh_config.py::access_summary` already expresses "all tailnet ports
  reachable at host:port" — reuse its text in the UI.

## Design

### Resolver: CoreDNS on the control node

`infra/coredns/Corefile`:

```
myhomecloud.dev {
    file /etc/coredns/db.myhomecloud.dev
    reload 5s
    log
    errors
}
. {
    forward . 1.1.1.1 9.9.9.9
    cache 30
}
```

Zone file `/etc/coredns/db.myhomecloud.dev` is **generated by the controller**. Template:

```
$ORIGIN myhomecloud.dev.
$TTL 60
@   IN SOA ns.myhomecloud.dev. admin.myhomecloud.dev. ( {serial} 7200 3600 1209600 60 )
@   IN NS  ns.myhomecloud.dev.
ns  IN A   {control_node_tailscale_ip}
{for each instance}
{instance}            IN A {instance_tailnet_ip}
*.{instance}         IN A {instance_tailnet_ip}     ; service subdomains resolve to same host
{end}
```

Note CoreDNS (private side) **can** use wildcards, so `*.app` → app's tailnet IP is fine and
covers `grafana.app`, `db.app`, etc. on the tailnet.

Add the coredns service to the control node `docker-compose.yml`:

```yaml
  coredns:
    image: coredns/coredns:latest
    restart: unless-stopped
    command: -conf /etc/coredns/Corefile
    volumes:
      - ./infra/coredns/Corefile:/etc/coredns/Corefile:ro
      - coredns_zones:/etc/coredns        # controller writes db.myhomecloud.dev here
    ports:
      - "53:53/udp"
      - "53:53/tcp"
```

### Controller code

Add `src/homecloud/dns/zone.py`:

- `render_zone(instances, control_node_ip) -> str` building the file above; serial =
  `YYYYMMDDNN` or a unix timestamp (must increase on every change).
- `write_zone()` reads instances from state, writes `settings.coredns_zone_path`, and (if set)
  runs `settings.coredns_reload_cmd` — but with `reload 5s` in the Corefile, no explicit reload
  is needed. No-op gracefully when the path's directory does not exist.
- Call `write_zone()` after any instance create/delete and after service changes.

### Tailscale split DNS (manual, document in phase output)

In the Tailscale admin console → **DNS**:
- Under **Nameservers**, "Add nameserver" → custom → control node tailnet IP.
- Enable **Restrict to domain** = `myhomecloud.dev` (this is the "split DNS" setting).
- Leave MagicDNS on. Now tailnet devices send `*.myhomecloud.dev` to CoreDNS on the control
  node; everything else uses their normal resolvers.

### Behavior to verify

- On the tailnet: `dig app.myhomecloud.dev` → instance tailnet IP (100.x). `psql -h
  db.app.myhomecloud.dev ...` connects over WireGuard, privately.
- Off the tailnet: `app.myhomecloud.dev` resolves via Cloudflare (web only, if published).
- A database with no published web service has **no** Cloudflare record and is therefore
  private-only.

## Acceptance criteria

- Creating/deleting an instance regenerates the zone file with a higher serial; CoreDNS serves
  the new records within ~5s.
- From a tailnet device, friendly names resolve to tailnet IPs and a TCP service (e.g. Postgres
  5432) is reachable by name.
- `write_zone()` no-ops cleanly when the zone dir is absent (dev).
- `uv run pytest -q` and `uv run ruff check src/` pass.

## Testing

- Unit-test `render_zone` (records present, serial monotonic, wildcard lines).
- Manual: split-DNS configured, `dig`/`psql` from a laptop on the tailnet.

## Out of scope

- Exposing databases publicly (intentionally not supported).
- Cloudflare records (Phase 03) — the private plane does not touch Cloudflare.
